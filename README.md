1)Eles funcionam como a "fonte única de verdade" do projeto. Além de servirem como um backup de segurança na nuvem, essas plataformas oferecem ferramentas essenciais que o Git sozinho não possui
2)O git pull é o comando de sincronização "de fora para dentro". Ele busca as alterações que seus colegas enviaram para o servidor e as mescla no seu diretório local

3)Fazer o push com frequência é muito mais do que uma etapa técnica; é um hábito que protege você e ajuda o time. Pense nisso por três ângulos:

Transparência com o time: Quando você sobe seu código, todo mundo consegue ver o que está sendo feito. Isso evita aquele problemão de duas pessoas mexerem na mesma coisa ao mesmo tempo sem saber, o que poupa muito estresse com conflitos depois.
Feedback rápido: Quanto antes você envia suas alterações, mais cedo as ferramentas de teste automático podem conferir se está tudo certo. É melhor descobrir um errinho agora do que acumular vários para o final da semana.
Sua "rede de segurança": Computador é máquina e, às vezes, resolve dar problema na pior hora possível. Se você fez o push, seu trabalho está salvo na nuvem. Se o seu notebook pifar, você não perde o dia e consegue retomar de onde parou em qualquer outra máquina.
